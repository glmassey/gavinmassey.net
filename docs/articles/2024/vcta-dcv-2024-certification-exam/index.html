Section 1 - Architecture and Technologies

Objective 1.1 - Identify how physical resources are presented to multiple virtual machines

Virtualization Overview

  • Virtualization allows multiple Virtual Machines (VMs) to run on a single physical host.
  • The hypervisor, such as VMware ESXi, is responsible for distributing physical resources to VMs.

CPU Virtualization

  • VMs are assigned virtual CPUs (vCPUs) which correspond to physical CPU cores or threads.
  • The hypervisor manages the scheduling of vCPUs on the physical CPUs.
  • Overcommitment of CPU resources is possible, where more vCPUs are allocated than available physical cores.

Memory Virtualization

  • Memory is dynamically allocated from the host’s physical memory to VMs.
  • Techniques like ballooning and swapping are used to manage memory efficiently.
  • Memory overcommitment involves allocating more memory to VMs than is physically present.

Storage Virtualization

  • VMs access storage through virtual disks, which are files on physical storage devices.
  • VMware File System (VMFS) allows shared storage access by multiple ESXi hosts.
  • Datastores are logical containers for storage, where VMs store their files like virtual disks and snapshots.

Network Virtualization

  • Virtual switches connect VMs to the physical network and function similarly to physical switches.
  • Port groups on virtual switches define network policies for VMs.

Resource Pools

  • Resource pools are logical abstractions for managing CPU and memory resources.
  • They allow the definition of CPU and memory reservations, limits, and shares for VMs.

Objective 1.2: Identify How Virtual Resources Can Be Shared Across Multiple Virtual Machines

Resource Types

  • Includes CPU, memory, power, storage, and network resources.
  • ESXi manages network bandwidth and disk resources using network traffic shaping and a proportional share mechanism.

Resource Providers

  • Hosts and clusters (including datastore clusters) provide physical resources.
  • For hosts, available resources are the hardware specification minus resources used by virtualization software.
  • A cluster is a group of hosts managed jointly by vCenter Server, owning all CPU and memory resources of the hosts.
  • Datastore clusters, groups of datastores, are managed jointly by vCenter Server with options like Storage DRS for load balancing and space utilization.

Resource Consumers

  • Virtual machines consume resources.
  • Default resource settings are generally sufficient, but can be adjusted for CPU, memory, and storage I/O.
  • Resource allocation involves share-based percentages or guaranteed reservations.
  • Admission control ensures enough unreserved resources are available for VM power-on.
  • Resource pools, acting as both providers and consumers, manage resources hierarchically.

Resource Allocation by ESXi Hosts

  • Allocation is based on user-defined limits, total host or cluster resources, VM numbers and usage, and virtualization overhead.

Goals of Resource Management

  • To resolve resource overcommitment and achieve performance isolation, efficient usage, and easy administration.
  • Performance isolation ensures VMs don’t monopolize resources.
  • Efficient usage aims to leverage undercommitted resources and handle overcommitment gracefully.
  • Easy administration involves controlling VM importance, dynamic partitioning, and fulfilling service-level agreements.

Objective 1.3: Identify Examples of Type 1 and Type 2 Hypervisors

Hypervisor Overview

  • A hypervisor, or virtual machine monitor (VMM), creates and runs VMs, sharing a host computer’s resources like memory and processing.

Type 1 Hypervisors (Bare Metal)

  • Run directly on the host’s hardware, acting like a lightweight operating system.
  • Provide high security and performance.
  • Common in enterprise data centers for efficient, secure computing needs.
  • Examples: VMware ESXi, Microsoft Hyper-V for Windows Server, Citrix XenServer.

Type 2 Hypervisors (Hosted)

  • Run as a software layer on an operating system.
  • Suitable for end users and software testing where higher latency is acceptable.
  • Enable running different OSs on top of the hypervisor.
  • Examples: VMware Workstation, Oracle VirtualBox, Parallels Desktop for Mac.

Comparison

  • Type 1 hypervisors offer better performance and security but are more complex to manage.
  • Type 2 hypervisors are easier to set up and use but have higher latency and lower performance.

Use in Cloud Computing

  • Hypervisors are essential in cloud environments, supporting multiple VMs and enabling cloud-based applications and services.
  • Drive innovation in cloud computing by facilitating VM migration and management across diverse hardware.

Hypervisor Benefits

  • Speed: Quick creation of VMs.
  • Efficiency: Better resource utilization on a single physical machine.
  • Flexibility: Allow OS and applications to run on various hardware types.
  • Portability: VMs independent of physical hardware, allowing workload shifts and resource reallocation as needed.

Objective 1.4: Identify Business Challenges Addressed by vSphere

Reducing Planned Downtime

  • Planned downtime, typically over 80% of data center downtime, is caused by hardware maintenance, server migration, and firmware updates.
  • vSphere minimizes the impact of planned downtime by allowing dynamic movement of workloads to different physical servers without downtime, thus:
    • Eliminating downtime for common maintenance operations.
    • Eliminating planned maintenance windows.
    • Enabling maintenance at any time without disrupting users and services.
  • vSphere vMotion and Storage vMotion enable workload movement without service interruption, allowing faster, transparent maintenance operations.

Preventing Unplanned Downtime

  • vSphere has built-in capabilities to protect against unplanned downtime caused by hardware or application failures, transparent to OS and applications in VMs.
  • Key availability features in vSphere include:
    • Shared storage for eliminating single points of failure.
    • Network interface teaming for network card failure tolerance.
    • Storage multipathing for storage path failure tolerance.
  • vSphere HA and Fault Tolerance features minimize or eliminate unplanned downtime, providing rapid recovery and continuous availability.

vSphere HA for Rapid Recovery

  • vSphere HA leverages a cluster of ESXi hosts for rapid recovery and high availability of applications in VMs.
  • Protects against server, application, datastore accessibility failures, and network isolation.
  • Integrates with vSphere Distributed Resource Scheduler (DRS) for failure protection and load balancing.
  • Benefits of vSphere HA include minimal setup, reduced hardware cost, and increased application availability.

vSphere Fault Tolerance for Continuous Availability

  • vSphere Fault Tolerance ensures no data loss, transaction interruption, or network connection loss during a host failure.
  • Maintains identical states of Primary and Secondary VMs for continuous availability.
  • Transparent failover and automatic redundancy re-establishment.

Protecting vCenter Server

  • vCenter High Availability protects against host, hardware, and vCenter Server application failures.
  • Offers automatic and manual configuration options for high availability with minimal downtime.
  • VMware Service Lifecycle Manager ensures vCenter Server availability by monitoring and restarting failed services.

Objective 1.5: Identify the Components of a vSphere Environment

vSphere Software Components

  • ESXi: The hypervisor that runs virtual machines, managing their resources and configurations. It provides bootstrapping, management, and other services for VMs.
  • vCenter Server: Central administration service for ESXi hosts, managing actions on VMs and hosts. Runs continuously for monitoring and management, even without direct user interaction.
  • vCenter Single Sign-On (SSO): Authentication service within vSphere, using secure token exchange for communication between vSphere components. Components include STS (Security Token Service) and Administration server.
  • vCenter Lookup Service: Maintains topology information of the vSphere infrastructure, enabling secure connections between vSphere components.
  • VMware Directory Service: Associated with the vsphere.local domain, it’s a directory service providing an LDAP directory on port 389. Supports multi-site mode for replicating directory service content across vCenter Single Sign-On nodes.
  • vCenter Server Plug-ins: Enhance vCenter Server functionality, typically with server and client components. They can alter the vSphere Client interface, adding new features and functionalities.
  • vCenter Server Database: Stores the status of VMs, hosts, and users managed in vCenter Server. Can be either remote or local to the vCenter Server system.
  • tcServer: Required for many vCenter Server web services. Features that need tcServer include CIM/Hardware Status tab, Performance charts, WebAccess, and others.
  • vCenter Server Agent: Installed on each managed host, collects and communicates actions from vCenter Server.
  • Host Agent: Part of the ESXi installation, this software collects, communicates, and executes actions through the vSphere Client.

Objective 1.6: Identify vSphere Virtual Networking Components and Types

Networking Concepts Overview

  • Physical Network: Network of physical machines connected for data exchange, where VMware ESXi runs.
  • Virtual Network: Network of VMs on a physical machine, connected logically for data exchange.
  • Opaque Network: Managed outside of vSphere, like VMware NSX networks, appearing as nsx.LogicalSwitch in vCenter Server.

Physical and Virtual Networking Components

  • Physical Ethernet Switch: Manages traffic between machines on a physical network, with multiple ports for connections.
  • vSphere Standard Switch: Similar to a physical Ethernet switch, connects VMs to each port and forwards traffic to the right VMs. Can connect to physical switches using uplink adapters.
  • vSphere Distributed Switch: Acts as a single switch across all hosts in a data center, centralizing virtual network provisioning, administration, and monitoring.

Additional vSphere Networking Elements

  • Host Proxy Switch: A standard switch on every host associated with a vSphere distributed switch, replicating its networking configuration.
  • Standard Port Group: Defines connection through a standard switch to the network, specifying port configuration options like bandwidth limits and VLAN tagging.
  • Distributed Port: A port on a vSphere distributed switch connecting to a host’s VMkernel or a VM’s network adapter.
  • Distributed Port Group: Associated with a vSphere distributed switch, it specifies port configuration options for each member port.
  • NSX Distributed Port Group: Part of the NSX virtual distributed switch, with port configuration managed using VMware NSX Manager or NSX API management tools.

NIC Teaming and VLAN

  • NIC Teaming: Involves multiple uplink adapters forming a team for load sharing or passive failover.
  • VLAN: Enables segmenting a physical LAN into isolated groups of ports, based on the 802.1Q standard.

VMkernel TCP/IP Networking Layer

  • Provides connectivity to hosts and handles vSphere vMotion, IP storage, Fault Tolerance, and vSAN traffic.
  • IP Storage: Uses TCP/IP for communication, like iSCSI and NFS, for VM datastores and direct .ISO file mounting.

Network Services in ESXi

  • Connects VMs to the physical network and each other.
  • Connects VMkernel services to the physical network.

VMware ESXi Dump Collector Support

  • Sends VMkernel memory state (core dump) to a network server during critical failures.
  • Supports both vSphere Standard and Distributed Switches.
  • Automatically updates IP address changes for the configured VMkernel adapter.

Objective 1.7: Identify the Characteristics of Storage Access Protocols for vSphere

iSCSI

  • Description: Presents block devices over a network, encapsulating SCSI commands in TCP/IP packets. Introduced in VMware ESX 3.0 in 2006.
  • Implementation Options: Network adapter with iSCSI capabilities, using software iSCSI initiator, dependent or independent hardware iSCSI initiator.
  • Performance Considerations: Can run over 1Gb or 10Gb networks, supports jumbo frames and multiple connections multiplexed into a single session.
  • Load Balancing: Round-robin path selection policy via VMware Pluggable Storage Architecture.
  • Resilience: Implements failover with Storage Array Type Plug-in.
  • Error Checking: Uses TCP for resending dropped packets.
  • Security: Implements CHAP for authentication; VLANs recommended.
  • VAAI Primitives: Supports a full complement of block primitives including Atomic test/set, Full copy, Block zero, Thin provisioning, UNMAP.
  • ESXi Boot from SAN: Yes
  • RDM Support: Yes
  • Maximum Device Size: 64TB

NFS

  • Description: Presents file devices over a network for mounting by ESXi hosts, using NFS version 3 over TCP/IP.
  • Implementation Options: Standard network adapter accessed using a VMkernel port.
  • Performance Considerations: Supports 1Gb or 10Gb networks and jumbo frames.
  • Load Balancing: No specific load balancing; configuration for aggregate bandwidth via multiple paths.
  • Resilience: Network adapter teaming for interface failover.
  • Error Checking: Uses TCP for resending dropped packets.
  • Security: VLANs or private networks recommended for traffic isolation.
  • VAAI Primitives: Varies by array, includes Full copy, Preallocated space, Cloned offload using native snapshots.
  • ESXi Boot from SAN: No
  • RDM Support: No
  • Maximum Device Size: Depends on NAS array/vendor specifications.

Fibre Channel (FC)

  • Description: Presents block devices over a network using block access protocol, encapsulating SCSI commands into FC frames.
  • Implementation Options: Requires a dedicated host bus adapter (HBA).
  • Performance Considerations: Supports various HBA speeds; uses buffer-to-buffer and end-to-end credits for a lossless network.
  • Load Balancing: Round-robin path selection policy via VMware Pluggable Storage Architecture.
  • Resilience: Failover implemented via Storage Array Type Plug-in.
  • Error Checking: Lossless network implementation.
  • Security: Supports concepts of a VSAN for isolation, zoning between hosts and FC targets.
  • VAAI Primitives: Full complement of block primitives available.
  • ESXi Boot from SAN: Yes
  • RDM Support: Yes
  • Maximum Device Size: 64TB

Fibre Channel over Ethernet (FCoE)

  • Description: Similar to FC, but uses Ethernet for transport, encapsulating SCSI commands into Ethernet frames.
  • Implementation Options: Hardware converged network adapter or network adapter with FCoE capabilities using software FCoE initiator.
  • Performance Considerations: Requires 10Gb Ethernet and jumbo frames.
  • Load Balancing: Round-robin path selection policy via VMware Pluggable Storage Architecture.
  • Resilience: Failover implemented via Storage Array Type Plug-in.
  • Error Checking: Requires a lossless network with a pause frame mechanism at times of congestion.
  • Security: Supports VSANs for isolation and zoning between hosts and FCoE targets.
  • VAAI Primitives: Full complement of block primitives available.
  • ESXi Boot from SAN: Software FCoE – No, Hardware FCoE – Yes
  • RDM Support: Yes
  • Maximum Device Size: 64TB

Objective 1.8: Identify the Characteristics of vSphere Storage Technologies

Traditional Storage Virtualization Models

  • Storage virtualization refers to the logical abstraction of physical storage resources and capacities.
  • ESXi provides host-level storage virtualization.
  • Traditional model includes local and networked storage, and ESXi and vCenter Server virtualization functionalities.

Local and Networked Storage

  • ESXi storage management begins with preallocated storage space on different systems.
  • Supports both local storage and networked storage.

Storage Area Networks (SAN)

  • A high-speed network connecting ESXi hosts to storage systems.
  • ESXi uses Fibre Channel or iSCSI protocols for SAN connectivity.

Fibre Channel (FC)

  • A storage protocol used by SAN to transfer data from ESXi hosts to shared storage.
  • Utilizes Fibre Channel host bus adapters (HBAs) for connectivity.

Internet SCSI (iSCSI)

  • A SAN transport using Ethernet connections between ESXi hosts and storage systems.
  • Utilizes hardware iSCSI adapters or software iSCSI initiators.

Storage Device or LUN

  • In ESXi, device and LUN typically refer to a storage volume presented to the host from a block storage system.

Virtual Disks

  • VMs on ESXi hosts use virtual disks for storing operating systems, applications, and other data.
  • Resides on a datastore deployed on physical storage.
  • Accessible to VMs as if they were SCSI drives connected to a SCSI controller.

VMware vSphere VMFS

  • The datastores on block storage devices use the VMFS format.
  • A high-performance file system optimized for storing VMs.

NFS

  • ESXi’s NFS client uses the NFS protocol over TCP/IP to access NFS volumes on a NAS server.
  • The ESXi host can mount the volume and use it as an NFS datastore.

Raw Device Mapping (RDM)

  • Allows a guest OS inside a VM to have direct access to a storage device.

Software-Defined Storage Models

  • Abstracts underlying storage capacities and capabilities from VMs.
  • Involves vSphere technologies like VMware vSphere Virtual Volumes (vVols) and VMware vSAN.

VMware vSphere Virtual Volumes (vVols)

  • Changes storage management from datastores to abstract storage objects controlled by storage arrays.
  • Individual VMs become units of storage management.

VMware vSAN

  • A distributed layer of software running as part of the hypervisor.
  • Aggregates local or direct-attached capacity devices of an ESXi host cluster into a single storage pool.

Storage Policy Based Management (SPBM)

  • A framework providing a control panel across various data services and storage solutions.
  • Aligns application demands of VMs with storage entity capabilities.

I/O Filters

  • Software components on ESXi hosts offering additional data services to VMs, like replication and encryption.

vSphere Storage APIs

  • Used by third-party providers to develop components enhancing vSphere features.
  • Includes APIs for Storage Awareness, Array Integration, and Multipathing.

Objective 1.9: Identify the Purposes of Different Virtual Machine Files

.vmx (vmname.vmx)

  • Usage: Virtual machine configuration file.
  • Description: Stores the configuration settings of the virtual machine, including hardware settings and how the VM interacts with the ESXi environment.

.vmxf (vmname.vmxf)

  • Usage: Additional virtual machine configuration files.
  • Description: Used for additional configuration information, particularly when a VM is part of a team or has multiple configurations.

.vmdk (vmname.vmdk)

  • Usage: Virtual disk characteristics.
  • Description: Describes the characteristics of a virtual disk; serves as a pointer to the actual disk data files.

-flat.vmdk (vmname-flat.vmdk)

  • Usage: Virtual machine data disk.
  • Description: Contains the actual raw disk data for the virtual machine.

.nvram (vmname.nvram or nvram)

  • Usage: Virtual machine BIOS or EFI configuration.
  • Description: Stores the state of the VM’s BIOS or EFI, including settings and boot configuration.

.vmsd (vmname.vmsd)

  • Usage: Virtual machine snapshots.
  • Description: Contains metadata about snapshots, including the state of each snapshot and the relationship between snapshots.

.vmsn (vmname.vmsn)

  • Usage: Virtual machine snapshot data file.
  • Description: Captures the state of a VM at the time of the snapshot, including the VM’s memory state and the virtual machine settings.

.vswp (vmname.vswp)

  • Usage: Virtual machine swap file.
  • Description: Used for swapping virtual machine memory to disk when the ESXi host is under memory pressure.

.vmss (vmname.vmss)

  • Usage: Virtual machine suspend file.
  • Description: Stores the state of a suspended virtual machine, including the contents of the VM’s memory and its execution state.

.log (vmware.log)

  • Usage: Current virtual machine log file.
  • Description: Records events related to the virtual machine’s operation, useful for troubleshooting.

-#.log (vmware-#.log)

  • Usage: Old virtual machine log files.
  • Description: Stores historical log data for the VM. The number indicates the sequence of the log files.

Additional Files

  • .hlog: A log file used by vCenter Server for tracking VM files needing removal after certain operations.
  • .vmtx: Created when converting a VM to a template; replaces the .vmx file and describes the template’s configuration.

Objective 1.11: Identify Use Cases for Virtual Machine Snapshots, Cloning, and Templates

Virtual Machine Snapshots

  • Purpose: Snapshots preserve the state and data of a virtual machine at a specific point in time.
  • Use Cases:
    • Testing software with unknown or potentially harmful effects.
    • Reverting to a known state during iterative processes like software updates.
    • Creating restoration points for linear or branching processes.
    • Maintaining a baseline state before making changes to a VM.
  • Behavior and Limitations:
    • Each snapshot creates delta disk files capturing changes since the last snapshot.
    • Snapshots can impact VM performance and are not suitable for long-term backups.
    • Not supported for VMs with certain disk types or configurations (e.g., raw disks, RDM in physical mode, bus sharing).
    • Snapshots are not recommended as a permanent solution for production VMs.
    • Can consume significant disk space and are vulnerable to hardware failures.

Virtual Machine Cloning

  • Purpose: Cloning creates an exact copy of a virtual machine, including its virtual disks and settings.
  • Use Cases:
    • Rapidly deploying identical VM environments for testing or development.
    • Creating VM backups before major changes or updates.
    • Efficiently deploying multiple instances of the same VM configuration.

Virtual Machine Templates

  • Purpose: Templates are master copies of VMs used to create new VMs with a standard configuration.
  • Use Cases:
    • Standardizing VM deployments with pre-configured settings and applications.
    • Streamlining the setup process for large-scale VM deployments.
    • Ensuring consistent configurations for VMs across the organization.

In summary, snapshots are ideal for short-term state preservation and quick rollbacks, cloning is used for creating identical VM copies, and templates provide a standardized and efficient way to deploy multiple VMs with similar configurations.

Objective 1.12: Functionality of vSphere vMotion and Storage vMotion Technology

vSphere vMotion

  • Purpose: vSphere vMotion allows for the live migration of virtual machines from one host to another without downtime.
  • Functionality:
    • Compute Resource Change: Migrates a VM to a new host while keeping the virtual disk on the same datastore.
    • Compute and Storage Change: Moves both the VM state and its disks to a new host and datastore. This is possible even in environments without shared storage.
    • Transferred State Information: Includes current memory content and all the information that defines and identifies the VM, such as BIOS, devices, CPU, MAC addresses, chipset states, and registers.
    • Stages in vMotion: The process involves verifying VM stability, copying VM state information to the target host, and resuming activities on the new host.
    • Error Handling: If errors occur during migration, the VM reverts to its original state and location.

Storage vMotion

  • Purpose: Allows migration of a VM and its disk files from one datastore to another while the VM is running.
  • Functionality:
    • Used for moving VMs off arrays for maintenance or upgrade, optimizing disks for performance, and transforming disk types to reclaim space.
    • Can change disk provisioning type during migration.
    • Renames VM files on the destination datastore to match the VM’s inventory name.
    • Uses:
      • Storage maintenance and reconfiguration without VM downtime.
      • Redistributing storage load to balance capacity or improve performance.
  • Requirements and Limitations:
    • Supports persistent mode disks or RDMs.
    • Cannot migrate VM disks greater than 2 TB from VMFS5 to VMFS3 datastores.
    • Requires appropriate licensing and host access to both source and target datastores.

Virtual Machine Snapshots

    • Creating backups before updates.
    • Efficient deployment of multiple VMs with the same configuration.

Templates

  • Purpose: Provides a master copy of a VM to create new VMs with standardized configurations.
  • Use Cases:
    • Standardizing VM deployments.
    • Streamlining setup for large-scale deployments.
    • Ensuring consistent VM configurations across an organization.

Objective 1.14: Characteristics of vSphere High Availability and Fault Tolerance

vSphere High Availability (HA)

  • Purpose: Provides rapid recovery from outages and high availability for applications running in virtual machines.
  • Functionality:
    • Automatically restarts virtual machines on other hosts within the cluster if a server failure occurs.
    • Monitors VMs and resets them in case of application failure.
    • Protects against datastore accessibility failures and network isolation.
    • Integrates with vSphere Distributed Resource Scheduler (DRS) for load balancing.
    • Minimal setup required, increases application availability.

vSphere Fault Tolerance (FT)

  • Purpose: Provides higher availability than HA by protecting VMs from host failures without data loss or service interruption.
  • Functionality:
    • Maintains identical states of Primary and Secondary VMs at any point in time.
    • Ensures immediate failover if the host running the Primary or Secondary VM fails.
    • Maintains network connections and transactions during failover.
    • Automatically re-establishes redundancy after a failover.

vCenter High Availability (vCenter HA)

  • Purpose: Protects against host and hardware failures as well as vCenter Server application failures.
  • Functionality:
    • Offers automatic and manual configuration options.
    • Supports high availability with minimal downtime.
    • Automatic option clones the Active node to the Passive and Witness nodes.
    • Manual option requires cloning and networking configuration by the administrator.

VMware Service Lifecycle Manager

  • Purpose: Provides availability for vCenter Server.
  • Functionality:
    • Restarts failed vCenter services.
    • Monitors health of services and takes remediation actions.

Objective 1.16: Functionality of VMware Distributed Resource Scheduler (DRS)

VMware Distributed Resource Scheduler (DRS) groups VMware ESXi hosts into resource clusters to manage the computing needs of different business units. It offers:

  • High Availability: Provides highly available resources to workloads.
  • Workload Balance: Balances workloads for optimal performance.
  • Resource Management: Allows scaling and managing computing resources without service disruption.

DRS is included in the vSphere Enterprise Plus package and enables:

  • Service Level Improvement: Guarantees appropriate resources to virtual machines.
  • Cluster Flexibility: Deploys new capacity to a cluster without service disruption.
  • Maintenance Support: Automatically migrates virtual machines during maintenance.
  • Infrastructure Monitoring: Monitors and manages more infrastructure per system administrator.

Combined with VMware vSphere Distributed Power Management (DPM), it optimizes power consumption within a cluster. DPM adjusts host power usage based on demand, cutting power and cooling costs during low utilization periods.

Technical Details of DRS

  • Initial Workload Placement: On powering a virtual machine in a cluster, DRS places it on a suitable host or generates a recommendation based on the chosen automation level.
  • Automated Load Balancing: Distributes virtual machine workloads across vSphere hosts in a cluster, optimizing performance.
  • Power Consumption Optimization: DPM and DRS together manage power consumption at cluster and host levels, dynamically adjusting to workload demands.
  • Cluster Maintenance: Facilitates the VMware vSphere Update Manager remediation process by optimizing the number of hosts entering maintenance mode.
  • Constraint Correction: Ensures distribution of virtual machines across vSphere cluster hosts complies with user-defined rules during host failures or maintenance.

Objective 1.17: Given a DRS score, identify the meaning

The DRS migration threshold in VMware vSphere Distributed Resource Scheduler (DRS) determines the aggressiveness of DRS in recommending VM migrations to balance resources in a cluster. The threshold has five settings, ranging from Conservative to Aggressive:

  • Conservative Setting: Generates only priority-one recommendations, which are mandatory.
  • Aggressive Setting: More frequent recommendations for migrations to improve VM happiness.

The DRS score is a measure of VM happiness or execution efficiency:

  • Each migration recommendation is computed using the VM happiness metric, displayed as the DRS Score in the cluster’s Summary tab.
  • The Cluster DRS Score is a weighted average of the VM DRS Scores of all powered-on VMs in the cluster.
  • The color of the gauge component’s filled section changes based on the score value, corresponding to the VM DRS Score histogram.
  • The histogram shows the percentage of VMs with a DRS Score in a specific range.
  • You can view VMs sorted by their DRS score in ascending order in the Monitor tab of the cluster.

Objective 1.18: Use Cases for Enhanced vMotion Compatibility (EVC)

Enhanced vMotion Compatibility (EVC) ensures vMotion compatibility among hosts in a cluster:

  • EVC makes sure all hosts present the same CPU feature set to VMs, even if actual CPUs differ, preventing vMotion failures due to CPU incompatibilities.
  • Configuring EVC involves setting all host processors in a cluster to present a feature set of a baseline processor (EVC mode).
  • EVC mode is based on the host with the smallest feature set and uses AMD-V Extended Migration or Intel FlexMigration technology to mask processor features.
  • EVC allows VMs to take advantage of faster processor speeds, more CPU cores, or hardware virtualization support available on newer hosts.
  • It doesn’t prevent VMs from accessing hidden CPU features in all circumstances; applications not following CPU vendor recommended methods of feature detection might behave unexpectedly.
  • From vSphere 7.0 Update 1 onwards, EVC can be used for Virtual Shared Graphics Acceleration (vSGA), allowing multiple VMs to share GPUs for 3D graphics acceleration.

Section 4 - Installing, Configuring, and Setup

Objective 4.1: Virtual Switch Configuration Options

vSphere Standard Switches

  • Overview: Standard switches provide network connectivity to hosts and virtual machines. They can bridge traffic internally between VMs in the same VLAN and link to external networks.
  • Configuration:
    • Physical NICs on hosts are connected to uplink ports on the standard switch.
    • Virtual machines have network adapters (vNICs) connected to port groups on the standard switch.
    • Each port group can use one or more physical NICs for network traffic.
    • If a port group does not have a physical NIC, VMs in the same port group can only communicate with each other but not with the external network.
  • Standard Port Groups:
    • Identified by a network label, unique to the current host.
    • Network labels help make the networking configuration of VMs portable across hosts.
    • Port groups connected to the same broadcast domain should have the same label.
    • VLAN ID is optional for restricting traffic to a logical Ethernet segment.
  • Number of Standard Ports:
    • The number of ports on standard switches dynamically scales up and down for efficient use of host resources.

vSphere Distributed Switch Architecture

  • Provides centralized management and monitoring of the networking configuration of all hosts associated with the switch.
  • Separates the data plane and the management plane.
  • Management functionality resides on the vCenter Server system, allowing network configuration at a data center level.
  • The data plane is locally on every host associated with the distributed switch, called a host proxy switch.
  • Introduces uplink port groups and distributed port groups for consistent networking configuration.

Editing vSphere Distributed Switch Settings

  • General and advanced settings control the switch-wide defaults, which can be overridden by port group settings.
  • General settings include the switch name and number of uplinks.
  • Advanced settings include Cisco Discovery Protocol settings and the maximum MTU for the switch.
  • Network I/O Control can be enabled or disabled.
  • MTU size can be adjusted, with options to enable jumbo frames by setting an MTU value greater than 1500 bytes.
  • Multicast filtering mode options include Basic and IGMP/MLD snooping.
  • Discovery Protocol settings allow for Cisco Discovery Protocol or Link Layer Discovery Protocol, with operation modes like Listen, Advertise, or Both.
  • Administrator contact details can be entered for the distributed switch.

Objective 4.2: Configuring Different Types of Datastores

Types of Datastores

  • VMFS (version 5 and 6): Deployed on block storage devices, optimized for virtual machines.
  • NFS (version 3 and 4.1): Accessed over TCP/IP from an NFS volume on a NAS server.
  • vSAN: Aggregates local capacity devices of hosts in a cluster into a shared datastore.
  • vVol: Represents storage containers in vCenter Server and vSphere Client.

Datastore Operations

  • Create Datastores: Different procedures for VMFS, NFS, and vVols datastores.
  • Administrative Operations: Include renaming, unmounting/mounting, removing datastores, and using datastore browser.
  • Change Datastore Name: Renaming possible without impacting running VMs.
  • Unmount/Mount Datastores: Remove datastore from visibility or re-enable visibility on specified hosts.
  • Remove VMFS Datastores: Delete datastores, destroying all files associated.
  • Datastore Browser: Manage datastore contents, upload/download files, move/copy/rename files, inflate thin disks.

Creating Datastores

  • VMFS Datastore: Set up on SCSI-based storage devices, extendable, and supports concurrent access.
  • NFS Datastore: Mount an NFS volume, with options for NFS 3 or 4.1.
  • vVols Datastore: Created using the New Datastore wizard, represents storage containers.

VMFS Datastores Characteristics

  • Supported Versions: VMFS5 and VMFS6, each with specific capabilities and limits.
  • Block Size: 1 MB for both VMFS5 and VMFS6.
  • Storage vMotion and Storage DRS: Supported across VMFS, vSAN, and vVols datastores.
  • Device Partition Formats: GPT for new datastores; VMFS5 upgraded from VMFS3 may use MBR.

Administrative Operations for Datastores

  • Rename, Unmount/Mount: Manage visibility and naming of datastores.
  • Remove VMFS Datastores: Permanently delete VMFS datastores and their contents.
  • Datastore Browser: Manage files and folders within datastores, perform various file operations.
  • Inflate Thin Virtual Disks: Convert thin-provisioned disks to thick format.
  • Storage Filters: Default filters in vCenter Server to avoid storage corruption.

Storage Filtering

  • Filters in vCenter Server help avoid unsupported use of storage devices and provide a consistent view of all VMFS datastores across all managed hosts. Filters include VMFS Filter, RDM Filter, Same Hosts and Transports Filter, and Host Rescan Filter.

Objective 4.3: Configuring vSphere HA

How vSphere HA Works

  • vSphere HA: Ensures high availability for virtual machines by pooling them with hosts into a cluster. In case of host failure, VMs are restarted on alternate hosts.
  • Primary and Secondary Hosts: The cluster elects one primary host to monitor all others. Secondary hosts run VMs and report to the primary.
  • Host Failure Types: vSphere HA identifies different types of host failures - Failure, Isolation, Partition. Responses depend on the failure type.

Creating a vSphere HA Cluster

  • Preparation: Requires a cluster with shared storage and network configurations among hosts.
  • vSphere HA Checklist: Includes ensuring all hosts are licensed, have static IP addresses, and share common management networks and storage.
  • Cluster Creation: Involve creating a cluster, adding hosts, and enabling vSphere HA with necessary configurations.

Configuring vSphere Availability Settings

  • Failures and Responses: Define responses to host failures, host isolation, VM monitoring, and VM Component Protection.
  • Admission Control: Specifies conditions under which VMs can be started, considering cluster resource availability.
  • Heartbeat Datastores: Select datastores for heartbeat mechanisms, helping to identify and respond to failures.
  • Advanced Options: Customize cluster behavior with specific advanced settings.

Detailed Configuration Aspects

  • Host Failure Responses: Control reactions to host failures, including VM restart priorities.
  • Host Isolation Responses: Define actions when a host loses network connectivity but continues running.
  • VMCP (VM Component Protection): Configure responses to data storage failures like PDL and APD.
  • VM Monitoring: Set monitoring levels to detect and respond to VM failures.
  • Proactive HA: Reacts to partial failures or degradations in host components.
  • vSphere HA Security: Includes secure communication, SSL certificate verification, and secure logging.

Customizing Individual VMs

  • VM Overrides: Allows customization of HA settings for specific VMs, diverging from cluster defaults.

Key Takeaways

  • vSphere HA is vital for ensuring high availability of VMs in a cluster.
  • Cluster Configuration: Proper setup of network and storage is crucial before enabling vSphere HA.
  • Customizability: vSphere HA offers extensive customization options to tailor the HA behavior as per specific requirements and scenarios.
  • Security Considerations: Secure communication and proper SSL configurations are essential for a reliable HA setup.

Objective 4.4: Configuring vSphere DRS

vSphere DRS Overview

  • vSphere DRS (Distributed Resource Scheduler): Essential for workload health in a vSphere Cluster. It relies on the availability of vCLS VMs from vSphere 7.0 Update 1.
  • vCLS VMs Dependency: If there are issues with vCLS VMs, DRS might display warnings on the Cluster Summary page. DRS settings and resource pools persist even if vCLS VMs quorum is lost.

DRS Cluster Requirements

  • Shared Storage: Utilize shared storage, typically on SAN or NAS.
  • Shared VMFS Volume: Place virtual machine disks on VMFS volumes accessible by both source and destination hosts.
  • Processor Compatibility: Maximize compatibility between source and destination hosts for vMotion transfers. Utilize Enhanced vMotion Compatibility (EVC) and CPU compatibility masks as needed.

vMotion Requirements for DRS Clusters

  • vMotion Network: Hosts in a DRS cluster should be part of a vMotion network for DRS migration recommendations.
  • Specific vMotion Configurations: Includes private Gigabit Ethernet migration network and unique network identity objects.

Creating a DRS Cluster

  1. Data Center: Verify existence in inventory.
  2. New Cluster: Right-click the data center and create a new cluster.
  3. DRS and HA Features: Select and configure necessary features.
  4. EVC Setting: Choose an EVC mode for CPU compatibility.

Editing Cluster Settings

  • Load Balancing: DRS monitors and balances CPU and memory usage.
  • Power Management: Utilizes DPM for efficient power usage.
  • Affinity Rules: Control VM placement within the cluster.

Steps to Edit Cluster Settings

  1. vSphere DRS: Navigate to the cluster and edit vSphere DRS settings.
  2. Automation Level: Choose between Manual, Partially Automated, and Fully Automated.
  3. Migration Threshold: Adjust DRS sensitivity.
  4. Predictive DRS: Optionally integrate with vRealize Operations.
  5. Additional Options: Customize based on cluster needs.
  6. Power Management: Set DPM automation level and threshold.

Custom Automation Level for Individual VMs

  • VM Overrides: Customize DRS automation levels for specific VMs in the cluster.

Disabling DRS

  • Turn Off DRS: Deselect the Turn On vSphere DRS option. This removes existing resource pools.

Restoring Resource Pool Tree

  • Restore Snapshot: Restore a previously saved resource pool tree on the same cluster with DRS enabled.

Key Takeaways

  • vSphere DRS: Critical for workload health, balancing resources, and managing power efficiently.
  • Dependency on vCLS VMs: Essential for DRS functionality in vSphere 7.0 Update 1 and later.
  • Customizability: Offers granular control over resource distribution and VM placement.
  • Compatibility and Networking: Ensure shared storage, VMFS volumes, and processor compatibility for effective DRS operation.

Objective 4.5: Configuring EVC

Enhanced vMotion Compatibility (EVC) Overview

  • Purpose: Ensures CPU compatibility between hosts in a cluster for seamless VM migration.
  • vSphere 7.0 Update 1 Feature: EVC now supports Virtual Shared Graphics Acceleration (vSGA), allowing VMs to share GPUs for 3D graphics acceleration.
  • Per-VM EVC: Enables migration across different processors and clusters, maintaining VM compatibility.

Cluster-Level vs Per-VM EVC

  • Per-VM EVC Modifications: Can only change when VM is powered off.
  • Cluster-Based EVC: A power cycle resets the VM’s EVC mode to the parent cluster/host’s mode.
  • Per-VM Overrides Cluster-Based EVC: If not configured, VM inherits parent EVC mode.
  • EVC Cluster Limitation: VM’s EVC mode can’t exceed the cluster’s EVC mode.

Compatibility and Requirements

  • Host Compatibility: ESXi 6.7 or later.
  • vCenter Server Compatibility: vCenter Server 6.7 or later.
  • VM Compatibility: Virtual hardware version 14 or later.

Configuring the EVC Mode of a Virtual Machine

  1. Power Off VM: Necessary to change EVC mode.
  2. Navigate to VM in vCenter Server: Access the Configure tab.
  3. VMware EVC Pane: Shows current EVC mode and details.
  4. Edit EVC Mode: Choose to enable/disable EVC for AMD or Intel hosts, or customize using API.
  5. Graphics Mode (vSGA): Choose a baseline graphics feature set.

Determining the EVC Mode of a Virtual Machine

  • EVC Mode Determination: At VM power-on, inheriting the cluster’s EVC mode.
  • EVC CPU Mode Column: Displays VM’s CPU mode in the VMs tab of a cluster or host.
  • EVC Graphics Mode (vSGA) Column: Shows baseline graphics features set for 3D graphics.

Key Takeaways

  • EVC Importance: Critical for seamless VM migration across various processors and cluster environments.
  • Per-VM EVC Feature: Provides flexibility in VM migration beyond the cluster level.
  • Compatibility Considerations: Requires specific ESXi and vCenter Server versions, and virtual hardware version.

This detailed explanation of EVC in vSphere 7.0 Update 1 highlights the critical role of EVC in maintaining VM compatibility across different processors and clusters, emphasizing the enhanced support for graphics acceleration and the flexibility offered by per-VM EVC settings.

Objective 7.1: Creating and Managing VM Snapshots

Taking Snapshots of a Virtual Machine

  • Purpose: Captures the settings, disk, and memory state of a VM at specific times.
  • Best Practices: Take snapshots when the VM is not undergoing heavy I/O load or communicating intensively over the network.
  • Types of Snapshots:
    • Memory Snapshots: Captures the live state of the VM. Useful for precise state capture, such as before a software upgrade.
    • Quiesced Snapshots: Utilizes VMware Tools to quiesce the file system, ensuring a consistent state. Ideal for automated or periodic backups.

Excluding Disks from Snapshots

  • Disk Mode Setting: Change to independent mode to exclude a disk from snapshots.
  • Prerequisites: VM must be powered off, and any existing snapshots deleted.
  • Procedure:
    1. Right-click VM → Edit Settings.
    2. Select Hard disk → Choose independent disk mode (Persistent or Nonpersistent).
    3. Confirm by clicking OK.

Taking a Snapshot

  • Procedure:
    1. Navigate to VM → Snapshots tab.
    2. Click Take Snapshot.
    3. Enter snapshot details (name, description).
    4. Choose options (memory capture, quiesce file system).
    5. Click Create.

Reverting a Snapshot

  • Effects: Returns VM to the state when the snapshot was taken.
  • Considerations: Disk changes revert to the snapshot point; independent disks remain unaffected.
  • Procedure:
    1. Go to VM → Snapshots tab.
    2. Navigate to desired snapshot.
    3. Click Revert.

Deleting a Snapshot

  • Types:
    • Delete: Removes a single snapshot, consolidating changes to the parent disk.
    • Delete All: Removes all snapshots, consolidating changes to the base disk.
  • Impact: Deleting snapshots can affect VM performance during consolidation.
  • Procedure:
    1. Go to VM → Snapshots tab.
    2. Choose Delete or Delete All based on requirement.
    3. Confirm deletion.

Consolidating Snapshots

  • Use: Combines redundant delta disks to improve VM performance and save space.
  • Indication for Need: “Consolidation is required” message.
  • Procedure:
    1. Navigate to VM → Snapshots tab.
    2. Click Consolidate.
    3. Confirm by clicking OK.
    4. Verify consolidation success in the Needs Consolidation column.

Managing Snapshots

  • Interface: Snapshot tab in the vSphere Client.
  • Features: View snapshot details, revert to snapshots, edit names/descriptions, delete snapshots.
  • Snapshot Tree: Shows all snapshots and the current state (represented by “You are here” pin).

This guide provides a comprehensive overview of creating, managing, and understanding the implications of VM snapshots in vSphere 7.0, highlighting the importance of snapshot types, how to exclude certain disks, and best practices for managing the snapshot lifecycle.

Objective 7.2: Managing VM Templates and Clones

Managing VM Templates

  • Editing Templates: Check out VM templates to make changes, and then check them back in.

VM Templates and OVF Templates Differences

Check In a Virtual Machine to a Template

  • Prerequisites: Ensure the VM is powered off or suspended.
  • Procedure:
    1. Navigate to the content library or vSphere Client inventory.

    3. Provide check-in notes and confirm the action.

  • Use Case: Discard a VM if updates are not required or unwanted.

Revert to a Previous Version of a Template

  • Procedure: Navigate to the VM template in the content library or vSphere Client inventory and revert to the desired previous version.

Delete a Previous Version of a VM Template

  • Procedure: Navigate to the VM template in the content library or vSphere Client inventory and delete the unwanted previous version.

Each of these considerations plays a crucial role in the effective provisioning and ongoing management of VMs in a vSphere environment. Proper planning and configuration according to these guidelines can lead to a more stable, efficient, and scalable virtual infrastructure.

Objective 7.5: Concepts of Role-Based User Management in vSphere

Role-based user management in VMware vSphere is an essential component for securing and efficiently managing a virtual environment. It involves assigning specific roles with defined privileges to users or groups. Here’s a detailed explanation based on vSphere Security Update 1 - VMware vSphere 7.0:

Using Roles to Assign Privileges

  • Roles: Predefined sets of privileges.
  • Privileges: Rights to perform actions and read properties.
  • Assignments: Pairing a user or group with a role, associated with an inventory object. Different roles can be assigned for different objects.
  • Example: Group Sales can have Virtual Machine User role on Pool A and Read-Only role on Pool B.

System Roles and Sample Roles

  • System Roles: Permanent and non-editable.
  • Sample Roles: Provided by VMware for common tasks. Can be cloned, modified, or removed.
  • Customization: It’s advisable to clone a sample role first before making changes.

Custom Roles in vCenter Server and ESXi

  • vCenter Server Custom Roles: Created using the vSphere Client for managing privilege sets.
  • ESXi Custom Roles: Created using CLI or VMware Host Client for individual hosts. Not accessible from vCenter Server when managing ESXi hosts through it.

Creating a Custom Role

  • Process: Involves logging into the vCenter Server, accessing the Roles section in Administration, and creating or cloning a role.
  • Privilege Selection: Choosing appropriate privileges for the role.
  • Propagation: Role changes made in one vCenter Server are propagated to others in the same Single Sign-On domain.

vCenter Server System Roles

  • Administrator Role: Full access and rights, including assigning privileges.
  • Read-Only Role: Viewing state and details of objects without modification rights.
  • No Access Role: No visibility or modification rights.
  • Other Roles: Including No Cryptography Administrator, Trusted Infrastructure Administrator, and more.

Best Practices for Roles and Permissions

  • Assign to Groups: Preferably assign roles to groups rather than individual users.
  • Minimum Necessary Privileges: Grant only required privileges for necessary tasks.
  • Use of Folders: Helpful in grouping objects for permissions management.
  • Root Level Caution: Be cautious when adding permissions at the root level.
  • No Access Role for Masking: Use No Access role to restrict visibility in certain areas.
  • License Propagation: License changes propagate across vCenter Server systems in the same domain.

Required Privileges for Common Tasks

  • Task-Specific Privileges: Many tasks require permissions on multiple objects. For example, creating a VM requires privileges on the destination folder, host, datastore, and network.
  • Propagating Permissions: Ensuring that permissions are set correctly at different hierarchy levels for successful task completion.

In summary, role-based user management in vSphere is a powerful tool for administrators to control and delegate access within the virtual environment. It allows precise control over what actions users and groups can perform, enhancing both security and operational efficiency.

Objective 7.6: Identifying Virtual Networking Issues Impacting vSphere

From “vSphere Networking Update 1 - VMware vSphere 7.0”:

Networking Best Practices

  1. Stable Connections:

    • Avoid setting connection limits and timeouts between vCenter Server, ESXi, and other products to prevent service interruptions.

  2. Network Isolation:

    • Isolate networks for host management, vSphere vMotion, vSphere FT, etc., for enhanced security and performance.

  3. Dedicated Physical NICs:

    • Use separate physical NICs for groups of VMs, or utilize Network I/O Control and traffic shaping for bandwidth guarantees.

  4. Separate Network Services:

    • Create separate standard or distributed switches for different network services, or use VLANs for separation on a single switch.

  5. vSphere vMotion Network:

    • Keep vMotion connection separate to handle memory transfer during migration.

  6. Adding and Removing Network Adapters:

    • Possible without affecting VMs or network services.

  7. Deploy Firewalls for Sensitive VMs:

    • Use firewalls in VMs for routing between virtual and physical networks.

  8. Use VMXNET 3 NICs:

    • For best performance in VMs.

  9. Consistency in Physical Network Connections:

    • Ensure physical adapters connected to the same vSphere switch are also connected to the same physical network.

  10. Consistent MTU Configuration:

  • Avoid connectivity issues by configuring the same MTU on all VMkernel adapters in a Distributed Switch.

Objective 7.7: Identifying Virtual Storage Issues Impacting vSphere

Best Practices for Fibre Channel Storage

  1. One VMFS Datastore Per LUN: Only place a single VMFS datastore on each LUN for optimal performance and management.
  2. Avoid Changing Path Policy: Unless fully understood, avoid altering the path policy set by the system.
  3. Documentation: Maintain thorough documentation of the SAN setup, including zoning, access control, and configuration details.
  4. Failure Planning: Prepare for failures by creating topology maps and ensuring redundancy.
  5. Fibre Channel HBA Installation: Install HBAs in the correct slots and balance the PCI bus load.
  6. Performance Monitoring: Familiarize with the various monitoring tools available in vSphere and ESXi for performance tracking.
  7. LUN ID Changes: Be cautious when changing LUN IDs as it can cause VMFS datastores to become inactive.

Disabling Automatic ESXi Host Registration

  • For manual control, disable automatic ESXi host registration with storage arrays by changing the Disk.EnableNaviReg parameter to 0.

Optimizing Fibre Channel SAN Storage Performance

  • LUN Assignment: Consider the diverse workload on a LUN used by multiple virtual machines.
  • Caching: Ensure read/write caching is enabled on the storage array.
  • Load Balancing: Regularly monitor and redistribute LUNs across Storage Processors (SPs) for balanced load.

Best Practices for iSCSI Storage

  1. Single VMFS on LUN: Assign only one VMFS datastore per LUN in an iSCSI SAN.
  2. Document and Plan: Keep detailed records and plans for SAN management.
  3. HBA Installation: Install iSCSI HBAs in the correct slots and balance load.
  4. iSCSI Name Uniqueness: Ensure unique and properly formatted iSCSI names for adapters.
  5. Network Performance: Monitor and manage the performance of the network connecting the SAN, focusing on avoiding oversubscription.

Network Performance

  • Avoid dropped packets and congestion by ensuring multiple input links are not leading to a single output link. Use additional links between switches or routers as needed.

Best Practices for Working with vVols

  1. Adherence to Guidelines: Follow VMware’s guidelines for the best experience with vVols.
  2. Storage Container Provisioning: Create storage containers based on specific requirements and avoid over-provisioning.
  3. vVols Performance: Use different VM storage policies for different virtual volume components and monitor I/O load on individual protocol endpoints.
  4. Array Limitations: Be aware of the limitations imposed by the array on the number of virtual volumes.
  5. VASA Provider Availability: Ensure the availability of the storage provider (VASA provider) and avoid placing it on vVols storage.

These best practices cover various aspects of managing and optimizing Fibre Channel and iSCSI storage in a vSphere environment, including configuration, documentation, load balancing, and monitoring. Additionally, when working with vVols, it’s crucial to understand the guidelines, limitations, and best practices for storage container provisioning and performance optimization.

Objective 7.8: Purpose of Monitoring Alarms, Tasks, and Events in vSphere

Events

  • Definition: Events are records of user or system actions on vCenter Server or host objects.
  • Examples: License key expiry, VM power-on, user login, host connection loss.
  • Event Data: Includes details like event generator, time, and type.
  • Types:
    • Error: Indicates a fatal problem, terminating the process.
    • Warning: Signals a potential risk that needs attention.
    • Information: Indicates successful completion of an operation.
    • Audit: Provides crucial audit log data for security purposes.

Alarms

  • Definition: Notifications triggered by events, conditions, or the state of an inventory object.
  • Elements:
    • Name/Description: Identifies and describes the alarm.
    • Targets: Defines the monitored object type.
    • Alarm Rules: Triggers, severity, and response operations.
    • Last Modified: Date and time of the last alarm modification.
  • Severity Levels: Normal (green), Warning (yellow), Alert (red).
  • Application: Alarms are specific to the object they are defined in and monitor specified inventory objects.

Objective 7.9: Monitoring vSphere Cluster and SDRS Cluster

Monitoring vSphere Cluster Services

  • Resource Monitoring: Monitor resources consumed by vCLS VMs.
  • Visibility: vCLS VMs appear in the VMs and Templates tab, not in the Hosts and Clusters tab.
  • Identification: vCLS VMs have a distinct icon and their purpose is described in the Summary tab.
  • Resource Allocation:
    • VMDK: 245 MB (thin disk)
    • Memory: 128 MB
    • CPU: 1 vCPU
    • Hard Disk: 2 GB
    • Storage on Datastore: 480 MB (thin disk)
  • Capacity Reservation: Each vCLS VM reserves 100MHz and 100MB in the cluster, with a maximum of 400 MHz and 400 MB for all vCLS VMs.
  • Health Status Monitoring: Available in the Cluster Services portlet in the Summary tab.
    • Healthy (Green): At least one vCLS VM running.
    • Degraded (Yellow): No vCLS VM running for less than 3 minutes.
    • Unhealthy (Red): No vCLS VM running for 3 minutes or more in a DRS-enabled cluster.

Monitoring Storage DRS (SDRS) Cluster

  • Storage DRS: Automates VM storage placement and load balancing.
  • Monitoring Tools: Use vCenter Server to monitor SDRS cluster performance and capacity.
  • Alerts and Alarms: Set up alerts for storage utilization, latency, and other SDRS-related metrics.
  • SDRS Recommendations: Review and manually apply or automate the application of SDRS recommendations for optimal storage performance.
  • Reports: Generate reports on storage usage, performance, and efficiency within the SDRS cluster.

Objective 7.10: Performing and Monitoring vMotion, Storage vMotion, and Cold Migrations

Migrating Virtual Machines

  • Types of Migration:
    • Cold Migration: Moving powered-off or suspended VMs to a new host or datastore.
    • Hot Migration (vMotion): Moving powered-on VMs to a new host without interruption.
    • Storage vMotion: Moving VM disk files to a different datastore while VM is running.
  • CPU Compatibility: Check for CPU compatibility during cold migrations.

Cold Migration

  • Process: Involves moving VM files to a new host/datastore with the VM powered off or suspended.
  • Network Traffic: Transferred via management network by default; can be redirected to provisioning TCP/IP stack for high volumes or long-distance migrations.

Migration with vMotion

  • Use Cases: Maintenance, load balancing, resource reallocation.
  • Requirements: Hosts must meet specific networking, storage, and CPU requirements.
  • Stages: Verification of VM state, state information transfer, resumption of VM on the new host.

Storage vMotion

  • Functionality: Allows VM disk migration without downtime, useful for storage maintenance and reconfiguration.
  • Requirements: VM disks must be in persistent mode or RDMs.

Additional Considerations

  • VM Network Compatibility: Ensure compatibility between source and destination networks.
  • MAC Address Management: Handle MAC addresses carefully to avoid duplication.
  • Swap File Location: Affects vMotion compatibility.
  • vMotion Without Shared Storage: Allows for cross-cluster and cross-datacenter migrations.
  • Migration Between vCenter Server Systems: Requires specific conditions, such as Enhanced Linked Mode and time synchronization.

Best Practices for Migration

  • Planning: Assess requirements and limitations for each migration type.
  • Monitoring: Track migration progress and performance impacts.
  • Troubleshooting: Address any errors or issues that arise during migration.
  • Documentation: Keep records of migrations for audit and review purposes.
